### 주요 개발 내용

- 앱의 최초 문제 정의를 잡았다.
  - 사용자가 설정한 시간과 간격에 따라 “지금/방금 무슨 생각을 했는지” 묻는다.
  - 사용자의 짧은 응답을 모아 일주일 단위로 편지처럼 정리한다.
  - 일기 작성 부담을 줄이고, 기록을 통해 자신을 객관적으로 보게 한다.
- 웹 MVP 방향을 먼저 잡았다.
  - 기록 입력
  - 감정 선택
  - 에너지 선택
  - 주간 요약/편지
  - 알림 설정
  - 캘린더 회고
- 서비스 이름과 톤의 초기 후보가 바뀌기 시작했다.
  - `Thought Letter`
  - `Dear.me`
  - `나에게`
  - 이후 `Log to Letter`, 최종적으로 `Log Planet` 방향으로 변화

### 시간대별 요청사항

- 초반
  - “너랑 앱 개발도 가능해?”에서 시작해, 앱 아이디어를 설명했다.
  - MVP를 바로 만들어달라는 요청으로 웹앱 제작 방향을 잡았다.
- 중반
  - 푸시 알림, 알림 간격, 방해금지 시간 등 앱의 핵심 루프를 정의했다.
  - “대신 일기를 써주고 나를 객관적으로 보게 하는 앱”이라는 핵심 가치를 정리했다.
- 후반
  - 유명 뇌과학/인지심리학 기반 제안은 MVP에서는 우선 구조만 남기고, 실제 AI/전문 지식 연결은 후속 단계로 미뤘다.

### 의사결정 나무

```text
기록 앱을 만들까?
└─ 긴 일기 앱은 부담스럽다
   └─ 짧은 순간 기록으로 간다
      ├─ 하루 1회 기록? → 부족함
      └─ 하루 여러 번 알림 → 채택
         └─ 기록을 그대로 보여줄까?
            ├─ 단순 리스트 → 보조 기능
            └─ 일주일 뒤 편지 → 핵심 경험
```

- AI 연결 없이 편지를 어떻게 생성할지 정해야 했다.

### 다음 할 일

  - 주간 기록이 5개 이상이거나
  - 기록 본문 합계가 100자 이상일 때
- 조건을 만족하지 못하면
  - AI를 호출하지 않음
  - 규칙 기반 fallback 편지를 사용
  - 요약 영역은 앱 내부 통계로 채움
- API key 관리
  - 앱 클라이언트에 넣지 않음
  - Supabase secret으로 저장

### 현재 개인정보 라벨 초안 판단

- 광고 없음
- 사용자 추적 없음
- 결제 없음
- 위치 권한 없음
- 연락처 권한 없음
- 사진 읽기 없음
- 사진 저장은 사용자가 편지 이미지를 앨범에 저장할 때만 사용
- 로그인 사용자의 이메일/User ID/사용자 생성 콘텐츠는 계정과 연결될 수 있음

### 출시 전 업데이트 필요 항목

- 앱 이름과 공개 문구는 `Log Planet`으로 업데이트했다.
- 편지보관함이 비활성화되고 행성 기능이 중심이 되었으므로 앱 내부 개인정보처리방침 초안의 편지 관련 표현은 최종 출시 전에 최신 기능 기준으로 한 번 더 정리한다.
- AI 기능이 실제 출시 빌드에 포함되는지 확정한다.
- 출시 빌드에서 사용하지 않는 마이크/음성 인식 권한은 제거했다.
- 계정 삭제와 사용자별 RLS 분리는 실제 테스트 계정으로 QA했다.
- 테스트 콘솔이 운영 빌드에서 노출되지 않는지 확인한다.
